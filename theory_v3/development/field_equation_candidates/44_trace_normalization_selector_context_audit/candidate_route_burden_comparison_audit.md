# Candidate Route Burden Comparison Audit

## Result

This script compared the burdens attached to metric choice, scale choice, parallel records, and continued deferral.

The main result is that route burdens are now visible enough to guide future planning, but burden comparison is not a selector. A route does not win merely because it has fewer obligations or cleaner bookkeeping.

## Main Findings

- Metric choice still needs object/scope clarity, `zeta` convention, traced dimension, an admissible selector or explicit convention record, and downstream caveats.
- Scale choice has the same class of burdens: object/scope clarity, `zeta` convention, traced dimension, selector/convention record, and downstream caveats.
- Parallel records require explicit labels, separated expressions, and a continuing ban on neutral-law collapse.
- Continued deferral is legitimate only if it names the missing information and the next narrowing route. Deferral without target risks becoming a loop.
- Smaller burden is context only. It cannot select a branch by itself.
- A burden map can recommend a later route class, but a recommendation is not execution.

## Boundary

Visibility is not closure. A listed obligation has not been satisfied merely because the script names it.

This script does not choose the route with the shortest burden list, does not complete trace normalization, and does not license Package B adoption or insertion.

## Steering Note

This is an anti-loop result. It says deferral can remain honest, but only when the next missing piece is named. It also implies that if the project wants to avoid premature convention choice, the next group should either build a real source hierarchy, make an explicit convention record, or choose parallel records as a deliberately labeled visibility route.
