# candidate_rejection_trigger_sieve — Result Note

## Result

The rejection sieve rejects forbidden adoption broadenings without rejecting the narrow conditional trace-normalization candidate itself.

This distinction is the main result: bad upgrades die, but the caveated audit candidate survives.

## Main Findings

The script identifies rejection triggers for any adoption route that would require:

```text
neutral F_zeta with an expression
unqualified B_s
numeric d leakage
recovery support
hidden branch choice
insertion as the only meaningful use
caveats acting as theorems
```

These are not merely cautions. They are route-killers for broadened adoption claims.

The narrow candidate survives only because it refuses those upgrades. It remains a conditional audit object with caveats attached. It is not adopted, not insertable, and not a safety theorem.

The script also blocks overreaction. Rejected broadenings do not imply a total no-go for the paired attempt, and survival of the paired attempt does not revive the rejected broadenings.

## Boundary

This sieve removes unsafe routes. It does not select a branch, prove an alternative, adopt the candidate, or license physical use.

## Steering Consequence

Future adoption language must preserve the rejection triggers. The narrow candidate can continue only as long as it remains separated from neutral collapse, numeric leakage, recovery construction, hidden branch choice, and field-equation insertion.
