# Candidate Membership Criterion Precondition Matrix

## Result

The script stated the membership criteria needed before `zeta_Bs -> T_zeta` can become testable in a later route.

Here, “membership testable” means only that the preconditions are explicit enough to know what a later theorem or declaration would need. It does not mean theorem-ready, declaration-ready, adopted, derived, insertable, or parent-ready.

## Main Findings

- The candidate object must be typed as a membership-test trace payload object.
- The payload must be trace-only and suitable for `T_zeta` safe trace-sector testing.
- Role-purity criteria must exclude residual, source, correction, boundary, divergence, recombination, insertion, active `O`, and parent roles.
- Branch-sensitive tests must keep `zeta_Bs_metric` and `zeta_Bs_scale` explicitly indexed.
- Diagnostic membership labels must remain inert.
- Mandatory exclusion zones must be named before any later declaration or theorem route.
- Declaration support remains a future route that requires closed object, sector, domain/codomain, criterion, role-purity, branch-status, and scope obligations.
- Theorem support requires actual derivation or proof beyond criterion visibility.

## Boundary

Criteria are preconditions only. They are not proofs, declarations, Package B adoption, insertion support, or parent readiness.

Role purity is not a source no-double-counting theorem. Branch-index consistency is not branch choice. Diagnostic labels cannot become active projectors.
