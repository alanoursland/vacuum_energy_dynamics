# Candidate Metric Branch Choice Readiness Ledger

## Result

This script inventoried what would be required before `B_s_metric` could be chosen as the trace-normalization branch.

It did not choose `B_s_metric`. It kept the metric branch as a future explicit-choice candidate only.

## Main Findings

The metric candidate form

```text
log(B_s_metric)=2*zeta/d
```

may remain visible for a later explicit choice record.

Before that choice could support a declaration, several burdens remain open: `B_s_metric` must have a clear metric-coefficient object, domain, scope, `zeta` convention, traced dimension `d`, and normalization scope. The factor-of-two burden must be explicit rather than hidden.

The script also clarifies what may count as context but not selection. Consequence audits may expose obligations, risks, and downstream burdens. They do not derive or select the metric branch.

## Boundary

The metric branch cannot be chosen from inherited notation, old overloaded `B_s` symbol shape, Schwarzschild recovery, `AB=1`, `B=1/A`, PPN gamma, weak-field success, or parent-fit convenience.

The candidate expression is not a declaration. A metric branch choice, even if later made, would still not license `B_s/F_zeta` insertion or parent-equation use.
