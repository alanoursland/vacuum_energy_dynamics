# Candidate Declaration Record Requirement Matrix — Result

## Result

This script states what a future parallel trace-normalization declaration-attempt record must contain. It does not write the declaration, but it makes the requirements explicit enough that the next group does not need to rediscover them.

## Main Findings

A future declaration-attempt record must identify itself clearly as an attempt, candidate, or conditional declaration route. It must not masquerade as an adopted postulate or field-equation insertion.

The paired branch domain must be retained. The record must keep `B_s_metric` and `b_s_scale` branch-indexed and noncollapsed, and it must list the two expressions separately:

```text
log(B_s_metric)=2*zeta/d
log(b_s_scale)=zeta/d
```

The future record must carry shared `zeta`, symbolic `d`, the numeric-d condition, an explicit status transition clause, and downstream caveats. Those caveats must keep no-insertion, no-active-`O`, no-safety-theorem, and no-parent boundaries attached even if the declaration attempt is allowed.

## Boundary

Requirement clarity is not declaration. The matrix does not install trace normalization, does not adopt Package B, and does not allow a caveat-free or single-expression declaration.

## Steering Conclusion

This result makes the next group more concrete. A declaration attempt is now well-specified enough to be attempted, but only if it includes all required fields and preserves the paired symbolic/caveated form.
